# Expected 997 — Structural Skeleton and Control-Number Echo Rules

> **Author:** Jayne (Tester / QA) · **Branch:** `feature/supplier-inbound-997-workflow`
> **Status:** Offline reference — NOT a live-verified artifact. Segment content is structurally
> correct per Simon's D-997-1 spec and the Build_997_Xml action in
> `logicapps/supplier/workflows/supplier-inbound-ack/workflow.json`.
> Control numbers (ISA13, GS06, ST02 in the 997 envelope) are **agreement-generated** at runtime
> and cannot be predicted offline. Only the **echo values** (AK102, AK202) are deterministic given
> a known input 850.

---

## 1. Happy-path 997 segment skeleton (single 850 accepted, 006030)

The supplier's `Build_997_Xml` action produces XML rooted at `X12_00603_997` in the BizTalk EDI
X12 namespace (`http://schemas.microsoft.com/BizTalk/EDI/X12/2006`). The X12 Encode action wraps
this with the ISA/GS envelope from the `Supplier-Purchaser-X12-997` send agreement.

```
ISA*00*          *00*          *ZZ*SUPPLIER01     *ZZ*PURCHASER01    *YYMMDD*HHMM*U*00603*{ISA13}*0*T*>~
GS*FA*SUPPLIER01*PURCHASER01*YYYYMMDD*HHMM*{GS06}*X*006030~
ST*997*{ST02}~
AK1*PO*{AK102}*006030~
AK2*850*{AK202}~
AK5*A~
AK9*A*1*1*1~
SE*6*{ST02}~
GE*1*{GS06}~
IEA*1*{ISA13}~
```

**Delimiters** (per Simon D-997-2, identical to the 850):
- Data element: `*` (42)
- Component element: `>` (62)
- Segment terminator: `~` (126)

**SE01 segment count = 6** (ST, AK1, AK2, AK5, AK9, SE — the six segments in the transaction set
body excluding ISA/GS envelope). This is the value the workflow hard-codes in `Build_997_Xml`.

## 2. Control-number echo rules (AK102, AK202) — deterministic from input 850

These values come **from the received 850** and must be present verbatim in the 997 body.
The supplier workflow reads them positionally from the flat-file output of `Decoded_850_FlatFile`
(star-split, tilde-split).

| 997 element | Source in received 850 | Workflow expression |
|-------------|------------------------|---------------------|
| `AK102` | GS06 (group control number) | `trim(split(split(outputs('Decoded_850_FlatFile'),'~')[1],'*')[6])` |
| `AK103` | GS08 (version = `006030`) | `trim(split(split(outputs('Decoded_850_FlatFile'),'~')[1],'*')[8])` |
| `AK202` | ST02 (transaction-set ctrl #) | `trim(split(split(outputs('Decoded_850_FlatFile'),'~')[2],'*')[2])` |

**Segment indexing rationale** (zero-based, tilde-split):
- Index `[0]` = ISA segment (interchange control header)
- Index `[1]` = GS segment (functional group header) → elements `[1]`..`[8]` are GS01..GS08
- Index `[2]` = ST segment (transaction set header) → elements `[2]` is ST02

This positional parsing is **UNVERIFIED** — the flat-file format must be confirmed at designer /
first-run. See the Runtime Health-Check in `docs/supplier-roundtrip-test-plan.md`.

### Example: fixture `purchase-order-3line.json`

If the purchaser's X12 send agreement generates:
- GS06 (group ctrl #) = `1`
- GS08 (version) = `006030`
- ST02 (transaction-set ctrl #) = `0001`

Then the 997 body must contain:
```
AK1*PO*1*006030~
AK2*850*0001~
```

The actual GS06 / ST02 values depend on the purchaser's agreement control-number counters and
**cannot be predicted offline**. Look them up in the purchaser send run's
`Encode_to_X12_850` action output at runtime.

## 3. Fixed (non-echo) 997 fields

| Field | Value | Rationale |
|-------|-------|-----------|
| `AK101` | `PO` | Functional group ID of the **850** functional group |
| `AK201` | `850` | Transaction set ID being acknowledged |
| `AK501` | `A` | Transaction set accepted (happy path) |
| `AK901` | `A` | Functional group accepted |
| `AK902` | `1` | # transaction sets included (GE01 of the 850 functional group) |
| `AK903` | `1` | # transaction sets received |
| `AK904` | `1` | # transaction sets accepted |

## 4. 997 envelope (agreement-generated — not deterministic offline)

The 997's **own** ISA13, GS06, and ST02 are **generated by the `Supplier-Purchaser-X12-997`
send agreement** in the supplier Integration Account. These are unrelated to the echoed AK102/AK202
values and increment independently on each transmission. You cannot predict them offline; look up
the actual values in the supplier run's `Encode_997_to_X12` action output.

Key envelope identities (per Simon D-997-2):
- ISA06 = `SUPPLIER01     ` (15-char space-padded)
- ISA08 = `PURCHASER01    ` (15-char space-padded)
- GS01 = `FA` (Functional Acknowledgment — NOT `PO`)
- GS02 = `SUPPLIER01`, GS03 = `PURCHASER01`
- ISA12 = `00603`, GS08 = `006030`

---

## 5. What this skeleton does NOT cover

- **Rejection / error path** (AK3/AK4 segments): out of scope for the happy-path fixture test.
  A rejected 850 would show `AK501=R`, `AK901=R`, `AK904=0`, plus AK3Loop1 detail.
- **Multiple transaction sets in one functional group**: the demo sends one 850 per interchange,
  so AK902/AK903/AK904 are always `1/1/1` and there is one `AK2Loop1`.
- **Purchaser-to-supplier TA1**: `ISA14=0` (no TA1 requested for this demo, per Simon D-997-2).
